OVERVIEW:
The daemon receives JSON-RPC requests from the ctl client over a Unix socket,
processes them through the Manager, and returns responses. This guide breaks
down the complete implementation flow with one-line descriptions per step.

================================================================================
SECTION 1: PROTOCOL DEFINITION (internal/protocol/jsonrpc.go)
================================================================================

1.1 Define RPCRequest struct
    Create a struct that holds method name, process name parameter, and raw params for flexibility.

1.2 Define RPCResponse struct
    Create a struct that holds success boolean, result data, error message, and optional metadata.

1.3 Define ProcessInfo struct
    Create a struct that holds pid, status string, uptime duration, retry count, last start time.

1.4 Add JSON marshaling/unmarshaling support
    Ensure RPCRequest/RPCResponse can be serialized/deserialized to/from JSON strings.

1.5 Define error codes enum
    Create constants for error types: InvalidMethod, ProcessNotFound, InvalidParams, OperationFailed, InternalError.

1.6 Add response builder helpers
    Create functions like NewSuccessResponse(data) and NewErrorResponse(code, message) for convenience.

================================================================================
SECTION 2: SOCKET LISTENER SETUP (cmd/daemon/main.go)
================================================================================

2.1 Import net package
    Add "net" to imports for Unix socket operations.

2.2 Define socket path constant
    Create const SOCKET_PATH = "/tmp/taskmaster.sock" at package level.

2.3 Remove existing socket file on startup
    Call os.Remove(SOCKET_PATH) to clean up any stale socket from previous runs.

2.4 Create Unix socket listener
    Call net.Listen("unix", SOCKET_PATH) and defer listener.Close() with error checking.

2.5 Change listener permissions
    Call os.Chmod(SOCKET_PATH, 0666) to allow ctl client to connect regardless of user.

2.6 Create connection accept loop in main
    Spawn a goroutine that runs listener.Accept() in an infinite loop until shutdown signal.

2.7 Handle listener errors gracefully
    If Accept() fails, log the error but continue accepting (except on "use of closed network connection").

2.8 Spawn worker goroutine per connection
    For each accepted connection, launch a new goroutine to handle that client's RPC calls.

2.9 Register shutdown cleanup
    On SIGTERM/SIGINT, close the listener and wait for all connection handlers to finish.

SECTION 3: CONNECTION HANDLER (cmd/daemon/main.go or internal/app/handler.go)
================================================================================

3.1 Create HandleConnection function signature
    Define func HandleConnection(conn net.Conn, manager *Manager, updates chan bus.ProcessUpdate) error.

3.2 Defer connection closure
    Ensure conn.Close() is called when the handler returns, regardless of success or error.

3.3 Set read deadline on connection
    Call conn.SetReadDeadline(time.Now().Add(30*time.Second)) to prevent hanging clients.

3.4 Create JSON decoder from connection
    Use json.NewDecoder(conn) to parse incoming JSON directly from the socket stream.

3.5 Decode incoming request
    Call decoder.Decode(&request) into an RPCRequest struct and handle decode errors.

3.6 Log incoming request
    Write to event bus or stderr: "RPC request received: method=%s, params=%s".

3.7 Validate request is not nil
    Return error if the decoded request is empty or has zero values.

3.8 Call request router
    Invoke a router function like RouteRPCRequest(request, manager) to dispatch to handler.

3.9 Handle routing errors
    Catch errors from the router (method not found, validation failed, etc.) and build error response.

3.10 Create JSON encoder to connection
     Use json.NewEncoder(conn) to write responses back to the client stream.

3.11 Encode response struct
     Call encoder.Encode(response) to marshal the response as JSON and send it back.

3.12 Log outgoing response
     Write to event bus or stderr: "RPC response sent: success=%v, error=%s".

3.13 Return gracefully
     Exit the handler function; connection will close via defer.

SECTION 4: REQUEST VALIDATION (cmd/daemon/main.go or internal/app/handler.go)
================================================================================

4.1 Validate method name is not empty
    Return InvalidMethod error if request.Method is "" or contains invalid characters.

4.2 Validate method name format
    Ensure method matches pattern "Taskmaster.MethodName" (check for prefix and dot).

4.3 Parse method name into action
    Extract the action part after the dot: "Taskmaster.Start" → "Start".

4.4 Validate action is a known RPC method
    Check against whitelist: Start, Stop, Restart, GetStatus, Reload, Shutdown (case-sensitive).

4.5 Parse process name from params
    Extract the "name" field from request.Params; it may be a single name or "all" keyword.

4.6 Validate process name format for single targets
    If name is not "all", verify it matches format "program:NN" with zero-padded digits.

4.7 Validate process name is not empty for required methods
    For Start/Stop/Restart/GetStatus, reject if name is "" (Reload/Shutdown don't need names).

4.8 Return comprehensive validation error
    If any validation fails, include the specific reason in the error response for debugging.

SECTION 5: REQUEST ROUTING (cmd/daemon/main.go or internal/app/router.go)
================================================================================

5.1 Create RouteRPCRequest function
    Define func RouteRPCRequest(req *RPCRequest, manager *Manager) (*RPCResponse, error).

5.2 Extract action from method name
    Parse "Taskmaster.Start" to get "Start" by splitting on dot and taking second part.

5.3 Switch on action string
    Use a switch statement with cases for each RPC method name.

5.4 Route to GetStatus handler
    Case "GetStatus": call handleGetStatus(req, manager).

5.5 Route to Start handler
    Case "Start": call handleStart(req, manager).

5.6 Route to Stop handler
    Case "Stop": call handleStop(req, manager).

5.7 Route to Restart handler
    Case "Restart": call handleRestart(req, manager).

5.8 Route to Reload handler
    Case "Reload": call handleReload(req, manager).

5.9 Route to Shutdown handler
    Case "Shutdown": call handleShutdown(req, manager).

5.10 Catch unknown methods
     Default case: return error response with "unknown method" message.

5.11 Return response from handler
     Each handler returns *RPCResponse; propagate it up through RouteRPCRequest.

SECTION 6: HANDLER FUNCTIONS - GetStatus (cmd/daemon/main.go or internal/app/handlers.go)
================================================================================

6.1 Create handleGetStatus function
    Define func handleGetStatus(req *RPCRequest, manager *Manager) (*RPCResponse, error).

6.2 Extract process name from request
    Get the "name" field from req.Params; handle "all" specially.

6.3 Lock Manager for reading
    Call manager.Mu.RLock() and defer manager.Mu.RUnlock() to safely read state.

6.4 Check if process exists (single target)
    If name != "all", verify that manager.Process[name] exists in the map.

6.5 Return ProcessNotFound error if missing
    If the named process does not exist, return error response with process name.

6.6 Retrieve ProcessInstance from manager
    Get pi := manager.Process[name] after verifying it exists.

6.7 Extract pid and status from ProcessInstance
    Call pi.GetPid() and pi.GetStatus() or read Pid and Status fields directly.

6.8 Calculate uptime duration
    Compute time.Since(pi.LastStart) to get how long the process has been running.

6.9 Build ProcessInfo struct
    Create a ProcessInfo with name, pid, status, uptime, retry count, last start time.

6.10 Handle "all" keyword
     If name == "all", iterate over all processes in manager.Process and build array.

6.11 Build success response
     Create response with ProcessInfo (or array of ProcessInfo) in the result field.

6.12 Return response
     Return response object and nil error to indicate success.

7.1 Create handleStart function
    Define func handleStart(req *RPCRequest, manager *Manager) (*RPCResponse, error).

7.2 Extract process name from request
    Get the "name" field from req.Params; reject if "all" (Start requires single target).

7.3 Validate single process name
    Return error if name is "all" or empty.

7.4 Lock Manager for reading
    Call manager.Mu.RLock() to check if process exists.

7.5 Verify process exists in manager
    Check if manager.Process[name] exists; unlock and return error if not.

7.6 Check current status before starting
    Read pi.GetStatus(); if already "running", return success (idempotent behavior).

7.7 Unlock for the read check
    Release the read lock before calling mutation methods.

7.8 Call Manager.Start method
    Invoke manager.Start(name) to attempt starting the process (Manager must have this method).

7.9 Handle start operation error
    If manager.Start returns error, wrap it and return error response with reason.

7.10 Log the start request
     Write to event bus: "RPC Start requested for %s", name.

7.11 Build success response
     Create response indicating process is starting; optionally include the pid.

7.12 Return response
     Return response object and nil error.

     Ensure one crashed handler doesn't prevent new connections from being accepted.

12.8 Handle broken pipe on write
     If encoder.Encode fails due to client disconnect, log but don't crash.

12.9 Validate Manager state before operations
     Check that manager, manager.Process, manager.Config are not nil before access.

12.10 Handle race conditions gracefully
      Use proper locking and validate state after lock acquisition (state may change between check and use).

SECTION 13: CONCURRENCY & SYNCHRONIZATION (cmd/daemon/main.go)
================================================================================

13.1 Create goroutine pool for connections
     Use sync.WaitGroup to track all connection handlers for clean shutdown.

13.2 Add each connection handler to WaitGroup
     Call wg.Add(1) before spawning goroutine, wg.Done() when handler exits.

13.3 Wait for all handlers on shutdown
     In main, after closing listener, call wg.Wait() to ensure all clients are done.

13.4 Use channels for shutdown coordination
     Create shutdownChan := make(chan struct{}) to signal all goroutines to stop.

13.5 Monitor shutdown channel in accept loop
     Use select with shutdownChan to break out of accept loop on shutdown signal.

13.6 Propagate shutdown to connection handlers
     Pass shutdownChan to handlers so they can detect daemon shutdown mid-request.

13.7 Close listener before waiting for handlers
     Ensure listener.Close() is called before wg.Wait() to prevent new connections.

13.8 Set timeouts on blocking operations
     Use context.WithTimeout on expensive operations to prevent indefinite blocking.

13.9 Implement graceful shutdown sequence
     1. Stop accepting new connections, 2. Allow in-flight requests to complete, 3. Stop processes, 4. Exit.

13.10 Log shutdown progress
      Write to event bus at each stage: "Closing listener", "Waiting for handlers", "Shutting down processes", "Daemon exiting".

================================================================================

15.1 Ensure Manager.Start(name string) exists

15.2 Ensure Manager.Stop(name string) exists
     Method must stop a single process; return error if process not found or already stopped.

15.3 Ensure Manager.Restart(name string) exists
     Method must stop then start a process; return error if either step fails.

15.4 Ensure Manager.Reload() exists
     Method must reload config from disk and apply diff (start new, stop removed, restart changed).

15.5 Ensure Manager.Shutdown() exists
     Method must stop all processes gracefully and signal daemon to exit.

     Method must return ProcessInfo with pid, status, uptime, retry count, and last start time.

15.7 Ensure Manager.GetAllProcessInfo() exists
     Method must return array of ProcessInfo for all managed processes.

15.8 Ensure Process mutations are thread-safe
     All Manager methods must use Mu.Lock()/Unlock() to protect concurrent access.

15.9 Ensure state is validated before mutation
     Check process exists, is in correct state, before attempting operation.

15.10 Ensure operations emit updates to bus
      All state changes must write ProcessUpdate to the updates channel for logging.

SECTION 19: PERFORMANCE & OPTIMIZATION
================================================================================

19.1 Use RWMutex instead of Mutex for Manager reads
     GetStatus is read-heavy; use RLock to allow concurrent readers.

19.2 Minimize lock hold time
     Don't hold locks during network I/O; only hold while reading/writing Manager state.

19.3 Batch multiple process operations if possible
     If GetAllProcessInfo called, iterate once with single lock, not per-process locks.

19.4 Consider connection pooling
     Future optimization: reuse connections instead of close/accept per request (not required now).

19.5 Avoid unnecessary struct copies
     Use pointers to ProcessInfo instead of copying large structs.

19.6 Pre-allocate response arrays
     When building array of ProcessInfo, use make([]ProcessInfo, 0, len(manager.Process)).

19.7 Use json.Encoder for streaming
     Already done: json.NewEncoder(conn) streams JSON without buffering entire response.

19.8 Set reasonable read/write timeouts
     Prevent slow clients from holding resources indefinitely.

19.9 Avoid logging in hot paths
     Don't log every single field; log at handler level, not inside tight loops.

19.10 Monitor goroutine count
       Keep track of active connection handlers; alert if count grows unexpectedly.
